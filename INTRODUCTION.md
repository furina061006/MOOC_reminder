# MOOC Reminder — 项目介绍

> 中国大学MOOC 作业提醒助手，再也不会漏交作业。

---

## 一句话

一个浏览器扩展，自动追踪你在 [中国大学MOOC](https://www.icourse163.org/) 上的所有未完成作业，在浏览器图标上显示数量，点开即可查看清单。

---

## 解决的问题

中国大学MOOC 的作业散落在不同课程的不同页面里，没有统一的"未完成作业"看板。你得挨个点进每门课去翻，费时费力，还容易漏。

这个扩展把所有课程的未完成作业汇总到一个面板里，一目了然。打开浏览器就能看到还剩多少没做，点一下就知道哪些要交了。

---

## 核心功能

| 功能 | 说明 |
|------|------|
| **图标数字** | 浏览器图标上直接显示未完成数量 |
| **颜色提醒** | 红色=过期了，橙色=48小时内截止，蓝色=还早 |
| **自动检测** | 测验、作业、考试全部自动检测完成状态，不用手动勾 |
| **跨课程汇总** | 所有课程的作业集中在一个面板里 |
| **截止提醒** | 到期前 48h、24h 弹桌面通知（可选） |
| **每日汇总** | 每天早上推送一条，告诉你今天有哪些要交（可选） |
| **静音课程** | 不感兴趣的课可以关掉，完全消失 |
| **跳转页面** | 点作业标题直接跳到 MOOC 上对应的页面 |
| **导出日历** | 一键导入手机日历 |
| **深色模式** | 自动适配系统的深色主题 |

## 如何工作

### 整个系统就像一个快递站

```
你（用户）
  ↓ 查看
看板（Popup）←———— 仓库管理员（Service Worker）
                        ↓ 收集
                   跑腿员（Content Script）→ MOOC 网站
```

**三个角色各干各的：**

#### 跑腿员（Content Script）
一个帮你站在 MOOC 网站里跑腿的小程序。你打开 icourse163.org 的任意页面时，它就悄悄跟着进来了。

它的活：
- 读你的**通行证**（浏览器 cookie 里的 `NTESSTUDYSI`，类似你的登录凭证）
- 去柜台点餐：发一个请求给 MOOC 的内部接口，说"把整门课的数据给我"
- 拿到一大包 JSON 数据（像一本厚厚的课程册子，里面列了所有章节、测验、作业、考试、分数）
- 把数据打包送回给仓库管理员

**跑腿员有两个版本：**
- 一个在 `/learn/` 页面上工作，除了跑腿还会"用眼睛看"页面上的内容（DOM 抓取，作为备用方案）
- 一个在所有 icourse163 页面上都跟着（哪怕你只是在看首页），专门负责跑腿

> 快问快答：为什么不用仓库管理员直接去拿数据？
>
> 试过了。仓库管理员（Service Worker）去敲门，MOOC 的保安（服务器）看他不是"店里的人"（origin 不对），直接拦住了（403 错误）。但跑腿员本来就站在店里，保安认识他，畅通无阻。

#### 仓库管理员（Service Worker）
一个坐在后台办公室里的人，浏览器一开他就在，但不是总醒着——浏览器看他没事做就会让他去睡觉（休眠），有活了再叫他。

他的活：
- 定时器响了 → 叫醒跑腿员去拿数据
- 收到跑腿员送回来的数据 → 拆开、整理、核对
  - **拆开**：从一大包 JSON 里找出每一条作业（标题、截止日期、类型、分数）
  - **判断类型**：接口返回的 `type` 字段说 2=测验、3=作业、6=考试
  - **判断完成没**：有分数？→ 完成了。提交过了？→ 算完成了。显示"已完成"文字？→ 算完成了
  - **判断互评阶段**：看 `evaluateStart/End` 和 `scorePubStatus` 字段
- 整理好的数据 → 存进仓库（`chrome.storage.local`，浏览器本地存储）
- 更新图标数字 → 更新 badge

> 他睡觉了怎么办？
>
> 第一次叫醒他可能有点慢，所以刷新时会重试最多 3 次。第一次他还在揉眼睛，第二次就清醒了。

#### 看板（Popup）
你点击浏览器图标后弹出的小面板。

它的活：
- 从仓库里读出数据，展示成列表
- 每门课分组，截止时间最近的在最前面
- 你勾选完成 → 告诉仓库管理员标记一下
- 你点标题 → 跳转到 MOOC 对应页面

### 数据是怎么从 MOOC 到你的看板的

```
你打开 icourse163.org 任意页面
  ↓
跑腿员跟着进来了
  ↓ 读 cookie 里的通行证
  ↓ 发 XHR（同域请求，跟你在页面上正常浏览一样）
  ↓ 接口 getLastLearnedMocTermDto.rpc 返回数据
  ↓ 把原始数据（~200KB JSON）打包发回后台
仓库管理员
  ↓ 拆包解析
  ↓ 跟已有的数据合并（去重、保留手动标记）
  ↓ 存进仓库
  ↓ 更新图标数字
你点图标
  ↓ 弹出看板 → 显示最新数据
```

### 为什么用 XHR 不用 fetch——一个曲折的故事

一开始我们让仓库管理员（Service Worker）直接去调 MOOC 接口。结果：

1. **原始接口 403** — 调 `getMocTermDto.rpc` 被拦住
2. **加前缀试试 200 但空** — 路径换成 `mm-course/` 前缀，返回 HTTP 200 但里面啥也没有
3. **换另一个格式 404** — DWR 协议也不行
4. **各种姿势：cookie 放 URL、cookie 放 body、加 Referer 头** — 全部 403

花了大半天查下来，根因是：**仓库管理员和 MOOC 网站不在一个"店里"**。

- 仓库管理员在 `chrome-extension://xxx`（扩展自己的世界）
- MOOC 网站在 `https://www.icourse163.org`
- 你去 MOOC 网站时浏览器会自动带上你的登录 cookie，但仓库管理员去敲门时，浏览器说"你谁啊？"——cookie 就不给了
- MOCC 的后端一看：没有 cookie、origin 不对 → 403 拒绝

后来参考了一个叫 GinsMooc 的开源项目（一个查 MOOC 答案的插件），发现它的做法是：**让跑腿员去拿，他本来就在店里**。

跑腿员（Content Script）在 `icourse163.org` 页面上运行，发请求时浏览器**自动带上**这个域名的 cookie，服务器的保安认识他，数据就回来了。

**教训**：不是接口有问题，是你从哪敲门的问题。

### 数据怎么合并才不乱

同一门课的数据可能有多个来源：
- 跑腿员从 API 拿的（主要来源）
- 跑腿员从页面上"看"到的（DOM 抓取，备用）
- 用户手动勾选的

合并规则很简单：

1. **用户手动标记的永远优先** — 你说完成了就是完成了，系统不能改回去
2. **API 标记完成的** — 不会被备用方案（DOM）覆盖回未完成
3. **新来的数据跟已有的对上号** — 通过课程 ID + 标题匹配，即使两次的 ID 不同也能认出是同一份作业

### 通知是怎么发的

**截止提醒** 每 5 分钟检查一次：

```
距离截止
  48 小时 → 发一次（跨过 48h 边界时）
  24 小时 → 发一次
  已过期 → 发一次
  其他时间 → 不发（已经发过的档位不再重复）
```

**每日摘要** 每天早上 8 点（可调）汇总接下来 48 小时内要交的作业，推送一条。如果 8 点没开电脑，开机时会自动补发。

- 纯浏览器端运行，**不需要服务器**

- **不会上传**任何个人信息

## 安装

1. 打开 `chrome://extensions/`
2. 开启右上角「开发者模式」
3. 点击「加载已解压的扩展」
4. 选择 `MOOC_reminder` 文件夹

> 未上架 Chrome 网上应用店，需通过开发者模式加载。

---

## 局限性

- ❌ **需要打开 MOOC 页面才能刷新数据** — 跑腿员得站在 MOOC 网站上才能干活。保持一个 MOOC 标签页开着就行，不限定在哪个页面
- ❌ **仅支持 中国大学MOOC** — 不支持其他平台，因为跑腿员只认识这一家的门
- ❌ **不跨设备同步** — 数据存在当前电脑的浏览器里，换电脑得重新抓
- ❌ **网站改版可能失效** — 如果 MOOC 换了接口或页面结构，可能需要更新适配
